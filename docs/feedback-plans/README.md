# Feedback Plans

Local implementation plans generated by `/triagato` from production feedback.

## How it works

1. Run `/triagato` (targets PROD) or `/triagato localhost` (targets local dev)
2. The command fetches open feedback, compares against existing plans, and creates new ones
3. Plans are stored in subfolders by hostname: `docs/feedback-plans/cleancentive.org/`, `docs/feedback-plans/localhost/`, etc.
4. Review the generated plans in the relevant subfolder
5. To implement a plan, tell your agent: "Implement the plan in docs/feedback-plans/{env}/{id}.md"
6. After implementation, update the plan's `status` to `implemented`

## Plan file format

Each `.md` file uses YAML frontmatter:

```yaml
---
feedback_id: <uuid>          # Links to PROD feedback item
title: <summary>              # Agent-generated short title
status: open | planned | implemented | wont-fix
category: bug | suggestion | question
severity: critical | high | medium | low
prod_status: new | acknowledged | in_progress | resolved
created: <ISO date>
last_synced: <ISO date>       # Last time triagato updated this from PROD
last_updated: <ISO date>      # Last time plan content changed
related: []                   # UUIDs of related feedback plans
---
```

## Authentication

triagato uses a device code flow to authenticate with PROD. On first run, it generates a 6-digit code and opens a browser page where you approve the login. The session token is saved to `infrastructure/.feedback-token` and reused across runs (365-day expiry).

## Directory structure

```
docs/feedback-plans/
  README.md                           ← this file (committed)
  cleancentive.org/               ← PROD plans (gitignored)
    {feedback-id}.md
  localhost/                           ← local dev plans (gitignored)
    {feedback-id}.md
```

## Notes

- Environment subfolders are **gitignored** — plans are local working documents
- triagato is **read-only** — it never modifies feedback status upstream
- Files are named `{feedback-id}.md` for easy cross-referencing
