## Analysis

The map at `/map` is rich (filters, viewport, heat metric, basemap) but all state lives in Zustand stores and is lost on reload. There is no way to share a specific view — the URL is just `/map`. The reporter is asking for URL-encoded state so the map becomes shareable / deep-linkable.

**Affected area:** frontend map page and its filter/map stores.

**Current state:**
- Filters (`datePreset`, `pickedUpFilter`, `myFilter`, `cleanupFilter`) live in [insightsFilterStore.ts](frontend/src/stores/insightsFilterStore.ts).
- `heatMetric` lives in [mapStore.ts](frontend/src/stores/mapStore.ts).
- Viewport (center, zoom) is local to MapPage and not persisted at all.
- Basemap theme is already persisted in localStorage (per-user preference, not part of the shared view).
- No existing URL-sync pattern — `useSearchParams` is only used for auth flows (token, device code).
- React Router v6.21.3 is in use.

**Root cause (gap, not bug):** the feature was never built. Filter store is route-agnostic; it has no concept of URL sync.

**Severity:** medium — quality-of-life suggestion, no broken functionality.

## Implementation Plan

### 1. Query-param schema (compact, omit defaults)

| Param | Source | Encoded value | Omit when |
|-------|--------|---------------|-----------|
| `since` | `datePreset` | `7d` / `30d` / `1y` | `all` |
| `picked` | `pickedUpFilter` | `picked` / `spotted` | `all` |
| `mine` | `myFilter` | `1` | `false` |
| `cleanup` | `cleanupFilter.cleanupId` | uuid | filter null |
| `cleanupDate` | `cleanupFilter.cleanupDateId` | uuid (only if `kind: 'date'`) | filter null or `kind: 'cleanup'` |
| `metric` | `heatMetric` | `mass` | `items` |
| `view` | viewport | `lon,lat,zoom` (3-decimal lon/lat, 2-decimal zoom) | no user interaction yet |

Basemap theme excluded — it's a per-user preference, not view content. `cleanupName` excluded — display-only; resolve from the cleanup list on hydration.

### 2. New helper: `frontend/src/lib/mapUrlState.ts`

Two pure functions:
- `serializeMapState({ filters, heatMetric, view }) → URLSearchParams`
- `parseMapState(searchParams) → { filters?, heatMetric?, view? }` — each field optional, invalid values silently dropped.

Single source of truth for keys + encoding.

### 3. Hydration on mount (read URL → stores)

In `MapPage`, one-shot effect that reads `useSearchParams()`, parses via `parseMapState`, and calls the matching setters on `useInsightsFilterStore` / `useMapStore`. If `view` is present, stash it in `initialViewRef` so the map-init effect uses it as initial `center`/`zoom` (instead of the hard-coded `[0,20]/zoom 2`) and set `userInteractedRef.current = true` + `hasFittedRef.current = true` so auto-fit-to-data doesn't override the deep-linked viewport.

Must run synchronously before map init to avoid a visible jump.

### 4. Write-back on change (state → URL)

Second effect in `MapPage` subscribes to the relevant store slices, computes new search params via `serializeMapState`, and calls `setSearchParams(next, { replace: true })` (replace so panning doesn't fill browser history).

Viewport gets its own effect bound to MapLibre's `moveend` (already debounced by the gesture).

Use a `hydratedRef` to suppress write-back during the initial hydration tick, so the incoming URL isn't immediately normalized/stripped.

### 5. Share button

Small floating control near the existing heat-metric control. Reuses [`useCopyToClipboard`](frontend/src/lib/useCopyToClipboard.ts) (same pattern as SpotDetail's copy-location).

- Label: "Share link" with a link icon; toggles to "Copied" for 2s.
- Click: copies `window.location.origin + '/map?' + serializeMapState(currentState)`.

### 6. Edge cases / non-goals

- URL sync is map-page-only — other routes (Insights, History) still rely on the in-memory store, unchanged.
- Active team not in URL; shared link viewed by a different-team user shows that user's team's data. Cross-team sharing out of scope.
- Auth: a shared link doesn't bypass auth; existing post-login redirect preserves `?...` query.
- Invalid params silently dropped → defaults apply.
