## Analysis
Three interlocking problems in cleanup-date handling:

1. **Naive UTC↔local conversion in the form.** [frontend/src/components/CleanupList.tsx:11-14](../../../frontend/src/components/CleanupList.tsx) and [CleanupDetail.tsx:37-40](../../../frontend/src/components/CleanupDetail.tsx) define `toDatetimeLocal()` that does `new Date(iso)` then extracts local hours via `getHours()`. The string assembled for `<input type="datetime-local">` mixes UTC and local incorrectly, so the user sees a shifted time after save/reload.

2. **Backend stores `timestamp` (no tz).** [backend/src/cleanup/cleanup-date.entity.ts:13-17](../../../backend/src/cleanup/cleanup-date.entity.ts) types `start_at`/`end_at` as `timestamp` (without time zone). Whatever string the client sends gets stored verbatim — the moment the user crosses tz contexts, meaning is lost.

3. **String comparison validation rejects valid earlier times.** [CleanupList.tsx:181-182](../../../frontend/src/components/CleanupList.tsx) sets the end-time `min` from `startAt || toDatetimeLocal(now)` and the onChange rejects `e.target.value < startAt` as a string comparison. After the timezone shift in (1), comparing the displayed strings can flip the order, so the user can't save.

4. **Filter window mismatches display.** [backend/src/cleanup/cleanup.service.ts:309-330](../../../backend/src/cleanup/cleanup.service.ts) `searchCleanups` decides ongoing/past/future from `start_at`/`end_at` vs `now`. Because storage is naive `timestamp`, the "ongoing" window the backend computes is offset from the time the user sees in the UI — so the cleanup only appears in the selector during the actual UTC window, not the local one.

## Implementation Plan
1. **Migrate storage to `timestamptz`.**
   - Alter `cleanup_date.start_at`, `cleanup_date.end_at` to `timestamptz` via a TypeORM migration. PostgreSQL will interpret existing values as the DB session's timezone — verify what that is on prod (`SHOW timezone`) before migrating to avoid silent shift.
   - If existing rows are wrong, do a one-shot data fix in the migration (e.g., assume `Europe/Zurich` and `AT TIME ZONE 'Europe/Zurich'`).
2. **Send and receive ISO-with-tz everywhere.**
   - Backend DTOs: validate as ISO 8601 with offset (zod `.datetime({ offset: true })` or class-validator equivalent).
   - Frontend: convert `<input type="datetime-local">` (which has no tz) to ISO with the user's local offset on submit. Drop the hand-rolled `toDatetimeLocal()` helper and use the standard `new Intl.DateTimeFormat`/`Date.toISOString()` round-trip instead.
3. **Fix earlier-time validation.**
   - Replace `e.target.value < startAt` string comparison with proper Date comparisons.
   - Set the `min` on the end-time input from a computed local Date, not the raw form string.
4. **Verify the cleanup-list filter** still picks the right "ongoing" rows after migration. Add a regression test covering 09:30 → 11:30 local creation that's visible during 10:00 local "now".
5. **End-to-end check on prod-like data:** create a cleanup at 10:00 local, reload, confirm it still shows 10:00; pick an earlier end (09:30) and confirm save is blocked correctly; advance the clock and confirm visibility window matches the displayed times.
