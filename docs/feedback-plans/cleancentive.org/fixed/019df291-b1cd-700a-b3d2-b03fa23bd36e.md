## Analysis

The literal request — let stewards edit picks — is a thin authorization change. But "picks" is ambiguous and the underlying community-effort framing points at a richer answer: open up **detected litter items** (the labels and weights inferred for each photo) to community editing, since that's where label quality and ML signal actually live.

**The codebase is already most of the way there for editing detected items:**

- `PATCH /spots/:spotId/items/:itemId` already requires only `JwtAuthGuard` — no ownership check ([backend/src/spot/spot.controller.ts:388](../../../backend/src/spot/spot.controller.ts), [spot.service.ts:392](../../../backend/src/spot/spot.service.ts)). Any logged-in user can already update any detected item's labels and weight on any spot.
- `detected_item_edits` already logs every field change with `field_changed`, `old_value`, `new_value`, `created_by`, `created_at` ([backend/src/spot/detected-item-edit.entity.ts](../../../backend/src/spot/detected-item-edit.entity.ts)).
- The `ItemEditor` UI ([frontend/src/components/SpotEditor.tsx:163](../../../frontend/src/components/SpotEditor.tsx)) does not gate by ownership client-side either.

**What's blocking the wiki experience:**

1. **No surface to non-owners.** `SpotEditor` is only mounted from `HistoryPanel` (your own picks). The only way other users see a spot is the map popup ([MapPage.tsx:178-196](../../../frontend/src/components/MapPage.tsx)) — a thumbnail with no edit affordance. There is no `SpotDetail` view anywhere.
2. **Add is owner-gated.** `addDetectedItem` checks `user_id === userId` ([spot.service.ts:460](../../../backend/src/spot/spot.service.ts)). A passerby can't say "you missed a bottle".
3. **Delete is owner-gated and would wipe history.** `deleteDetectedItem` does a hard `repository.remove()` ([spot.service.ts:534-550](../../../backend/src/spot/spot.service.ts)) and writes no edit-log row. Worse, `detected_item_edits.detected_item_id` has `onDelete: 'CASCADE'` ([detected-item-edit.entity.ts:13](../../../backend/src/spot/detected-item-edit.entity.ts)) — deleting the item today **also wipes its entire edit history**, foreclosing any future restore.

## Implementation Plan

### Phase 1 — Backend: open edit and make delete history-preserving

1. **Lift ownership on `addDetectedItem`.**
   - In [spot.service.ts:460](../../../backend/src/spot/spot.service.ts), change the lookup from `{ id: spotId, user_id: userId }` to `{ id: spotId }`. The `JwtAuthGuard` on the controller still requires authentication — `created_by` on the audit row pins identity.
   - Continue writing the `detected_item_edits` rows that are already there (`addDetectedItem` lines 489-507).

2. **Make `detected_item_edits` survive deletion of its parent.**
   - Migration: change the FK on `detected_item_edits.detected_item_id` from `ON DELETE CASCADE` to `ON DELETE SET NULL`. Make the column nullable.
   - Update [detected-item-edit.entity.ts:10-14](../../../backend/src/spot/detected-item-edit.entity.ts): `@Column('uuid', { nullable: true })` and `@ManyToOne('DetectedItem', { onDelete: 'SET NULL' })`.
   - This is the load-bearing change for the user's "don't lose historic data" requirement. After it, deletion can be hard, history persists.

3. **Lift ownership on `deleteDetectedItem` and write a tombstone audit row.**
   - In [spot.service.ts:534](../../../backend/src/spot/spot.service.ts), drop the `user_id === userId` check; require only `JwtAuthGuard` (already on the route).
   - **Before** removing the row, write per-field audit entries capturing the final state — one per non-null field (`object_label_id`, `material_label_id`, `brand_label_id`, `weight_grams`) with `old_value: <current>`, `new_value: null` — plus one tombstone row `field_changed: 'deleted'`, `old_value: null`, `new_value: null`. This matches the existing per-field pattern in `addDetectedItem`/`updateDetectedItem` and gives a future restore enough to reconstruct the item.
   - Then `repository.remove()` as today. The tombstone rows survive thanks to the FK change in step 2 (their `detected_item_id` becomes NULL).

4. **(Deferred) Restore endpoint.** Not in scope for this plan, but the audit-row design above keeps it cheap: `POST /spots/:spotId/items/restore` that takes a deleted-item id (still discoverable from the surviving audit rows pre-NULL'ing — see note below), reads the most recent per-field `new_value` chain plus the tombstone, and re-inserts. Note: once `detected_item_id` is set NULL on delete, you'd need to either (a) record the deleted id in the tombstone row's `old_value`/`new_value` so it stays queryable, or (b) keep the id but switch the FK to a soft `RESTRICT` and not actually delete. Going with (a) is simpler — write the deleted item's id into the tombstone row's `old_value` before nulling. Worth doing now even without the restore endpoint, since it costs nothing.

### Phase 2 — Frontend: expose the wiki surface

5. **New `SpotDetail` view, reachable from the map popup and elsewhere.**
   - Component `SpotDetail.tsx`: shows the photo, capture metadata, item list, and item edit/add/delete controls.
   - Wire from [MapPage.tsx:178-196](../../../frontend/src/components/MapPage.tsx) — popup gains an "Open spot" link that navigates to `/spots/:id` (introduce a route — currently the app is route-light; check `App.tsx` for the current shell).
   - Reuse the existing `ItemEditor` from [SpotEditor.tsx](../../../frontend/src/components/SpotEditor.tsx); extract it to its own file if needed. Don't duplicate the PATCH call site.

6. **"Add detected item" affordance for non-owners.** Already in `SpotEditor` for owners — same component, just rendered in `SpotDetail`.

7. **Confirm-on-delete dialog.** Hard-delete is now broadly available; pair it with a confirmation dialog ("Remove this item? An edit history row will be kept.") and a small note on what's preserved.

8. **History view.** Small "View edit history" toggle on `SpotDetail` that lists `detected_item_edits` rows for that spot's items, including tombstones. Backend endpoint to add: `GET /spots/:spotId/edit-history`.

### Phase 3 — Guardrails

9. **Authenticated edits only.** Keep `JwtAuthGuard` on every detected-item write endpoint. Don't open this to guest IDs — `created_by` traceability collapses without a stable identity.
10. **Per-user rate limit** on detected-item writes (e.g., 60/hour). The codebase doesn't appear to have a generic rate limiter — start with a simple in-memory counter scoped per user, or `@nestjs/throttler` if it's already a dep. Defer if not.
