## Analysis
A litter-item **brand** (Feldschlösschen, a Swiss brewery) was stored with an erroneous capital `S` mid-word: `FeldschlösSchen`. Two issues: the bad data, and the latent code bug that produced it.

**Root cause** — `titleCase` in `worker/src/detection.ts:17-19`:

```ts
function titleCase(name: string): string {
  return name.replace(/\b\w/g, (c) => c.toUpperCase());
}
```

`\b\w` is ASCII-only (no `u` flag), so `ö` is not a word char. In `Feldschlösschen` the `ö` acts as a word boundary and the following `s` matches `\b\w` and is uppercased → `FeldschlösSchen`. Applied when a new brand label is created (`detection.ts:71`). Lookups compare on `LOWER(name)` (`detection.ts:44,77`), so once the mangled label exists it is reused for all later detections of the same brand. The same regex also lives in the already-applied migration `1775000000000-MigrateTextToLabels.ts:293` — left untouched (a ran migration must not be edited).

**Data model** — `detected_items.brand_label_id` → `labels` (`type='brand'`) → `label_translations.name` (locale `'en'`). The brand string is stored once on the shared label; correcting that one row fixes every item referencing it.

**Severity** — a bug with no stack trace is "high" by the triage rubric, but real impact is a cosmetic single label; recorded as `medium`.

## Implementation Plan

### 1. Fix `titleCase` to be Unicode-aware — `worker/src/detection.ts:17-19`
```ts
export function titleCase(name: string): string {
  return name.replace(/(^|[^\p{L}])(\p{L})/gu, (_, sep, first) => sep + first.toUpperCase());
}
```
Capitalizes the first letter of each word (word = run of Unicode letters split on non-letters); never breaks inside an accented word: `feldschlösschen → Feldschlösschen`, `coca cola → Coca Cola`. Export for testing. Does not down-case, so it won't repair already-stored values — step 2 does that. Ships via normal repo/deploy.

### 2. Correct existing prod data — one-off SSH SQL, all label types
Decided explicitly (conflicts with the standing no-manual-server-edits rule; chosen for this one-off cosmetic correction — recurrence is prevented by step 1). **Run only after step 1 is deployed.** SSH to prod → `docker compose exec -T postgres psql -U <user> <db>`.

a. Audit (read-only, human-reviewed) — sweeps all label types:
```sql
SELECT lt.id, l.type, lt.name
FROM label_translations lt
JOIN labels l ON l.id = lt.label_id
WHERE lt.name ~ '[^[:ascii:]][A-Z]'   -- non-ASCII letter immediately followed by uppercase ASCII
ORDER BY l.type, lt.name;
```
Matches the bug's harmful firing only: `McDonald's` and `Coca-Cola` do **not** match; `FeldschlösSchen` does.

b. Repair — targeted UPDATE per enumerated row, human-verified casing:
```sql
UPDATE label_translations SET name = 'Feldschlösschen', updated_at = NOW() WHERE id = '<uuid-from-audit>';
```
Each UPDATE corrects every item referencing the shared label — no per-item work. Re-run the audit afterwards; it should return zero rows.

c. Record the exact statements run here once executed (no automated reproducibility on this path).

### 3. Unit test — `worker/src/detection.spec.ts`
Add `titleCase` cases: `feldschlösschen → Feldschlösschen`, `coca cola → Coca Cola`, and a guard that a lowercase input is not re-mangled into `FeldschlösSchen`.
