## Analysis
The "my picks" list is a hard-capped 50 items.

- Frontend [frontend/src/components/HistoryPanel.tsx:190-201](../../../frontend/src/components/HistoryPanel.tsx) hardcodes `limit: '50'` with no offset, cursor, or load-more UI.
- Backend [backend/src/spot/spot.controller.ts:272-303](../../../backend/src/spot/spot.controller.ts) `GET /spots` accepts `limit` (default 20, max 100) but **no offset/cursor**. The service [backend/src/spot/spot.service.ts:295-320](../../../backend/src/spot/spot.service.ts) calls `.take(limit)` ordered by `captured_at DESC` with no `.skip()`.

Result: anything older than the 50th most recent pick is unreachable. Once cleanups grow, this gets worse.

## Implementation Plan

**Approach:** Cursor pagination keyed on `(captured_at, id)`, not offset. Stable under concurrent inserts and natural for append-on-load-more. The feedback service uses offset/page pagination, but the spot list is mutable from the top (new uploads), so offset would shift items.

Cursor format: opaque `${capturedAtISO}|${id}` (no base64 — already opaque, easier to debug).

Response shape stays backwards-compatible: `{ spots, nextCursor }`. Existing callers ignoring `nextCursor` keep working.

### 1. Backend — `listSpotsWithFilters`

In [backend/src/spot/spot.service.ts](../../../backend/src/spot/spot.service.ts):

- Extend filters with `before?: string`.
- Add `.addOrderBy('spot.id', 'DESC')` after the existing `captured_at DESC` order.
- Use `.take(limit + 1)` to detect "has more".
- When `before` is set and parses cleanly, add `WHERE (spot.captured_at, spot.id) < (:beforeAt, :beforeId)`.
- Return `{ items, nextCursor }`. `nextCursor` is the last item's `captured_at|id` joined, or `null`.
- Update `listSpotsForUser` and `listSpotsForGuest` return types accordingly.

### 2. Backend — controller

In [backend/src/spot/spot.controller.ts](../../../backend/src/spot/spot.controller.ts):

- Accept `@Query('before') beforeQuery?: string`. Validate cheaply (`includes('|')`).
- Pass through to the service.
- Return `{ spots: SpotDto[], nextCursor: string | null }`.

### 3. Frontend — HistoryPanel

In [frontend/src/components/HistoryPanel.tsx](../../../frontend/src/components/HistoryPanel.tsx):

- Add `nextCursor` state.
- `loadHistory(cursor?)`: when cursor omitted, replace `reports`; when set, append.
- Build URL with `params.set('before', cursor)` when present.
- After fetch, set `nextCursor` from payload.
- `loadMore = () => loadHistory(nextCursor ?? undefined)`.
- Render an explicit "Load more" `<button>` below the list when `nextCursor !== null`. Disabled / "Loading..." while `isLoading`. **No infinite scroll.**
- Filter changes (`pickedUpFilter`, `datePreset`) must reset cursor and replace the list — done by routing the existing `loadHistory()` call without a cursor.
- Page size stays at 50.

### 4. Test

`backend/src/spot/spot.service.spec.ts` (new, `bun:test` per existing pattern in [backend/src/swagger/swagger-config.spec.ts](../../../backend/src/swagger/swagger-config.spec.ts)):

- Cursor consistency: page 1 returns `limit` items + cursor; insert a newer spot; page 2 with the cursor returns the original next batch with no overlap and no gap.
- If full repository test infra doesn't exist yet, fall back to a focused unit test that mocks `createQueryBuilder` and asserts the WHERE clause + `take(limit + 1)` behavior.

### 5. Manual verification

Test data approach: rather than seeding 60 dummy spots, **temporarily set `limit: '3'` in [HistoryPanel.tsx:215](../../../frontend/src/components/HistoryPanel.tsx#L215) during dev** so the existing local picks for `matthias.cullmann@gmail.com` already span multiple pages. Pagination logic is identical regardless of page size. Revert to `'50'` before commit.
