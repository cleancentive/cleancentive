## Analysis

The "My" chip is enabled on both `/map` and `/insights` ([ContextBar.tsx:44-52](frontend/src/components/ContextBar.tsx#L44-L52)). Both pages send `user_id=<me>` to the backend; both have the same root-cause issue — cleanup-related output is computed from the user's *spots* instead of from the user's *participation*.

**Map (`/map`):** The map renders two layers via `getMapData()` ([insights.service.ts:75-148](backend/src/insights/insights.service.ts#L75-L148)):
- Spot markers — already filtered by `user_id`, `since`, `cleanup_date_id`.
- Cleanup-date markers — completely unfiltered. The cleanups query joins `LEFT JOIN spots s ... ${and}`, but `${and}` only narrows the **spot** side of the LEFT JOIN, so it controls the `spot_count` aggregate but never excludes a `cleanup_date` row. Result: every cleanup location stays on the map regardless of "My", date preset, or active-cleanup context.

**Insights (`/insights`):** `computeFilteredStats()` derives every cleanup figure from spots:
- `totalCleanups`: distinct `cleanup_id` from the user's spots ([insights.service.ts:352-356](backend/src/insights/insights.service.ts#L352-L356)) → "Cleanups" headline ([InsightsPage.tsx:99](frontend/src/components/InsightsPage.tsx#L99)).
- `cleanupsTimeSeries`: per-week distinct `cleanup_id` from spots ([insights.service.ts:390-396](backend/src/insights/insights.service.ts#L390-L396)) → cleanups bar chart ([InsightsPage.tsx:142](frontend/src/components/InsightsPage.tsx#L142)).

A cleanup the user joined but didn't log a pick in is invisible on both pages.

**Schema:** `Cleanup` has no `user_id` ([cleanup.entity.ts](backend/src/cleanup/cleanup.entity.ts)); ownership/participation lives in `cleanup_participants` ([cleanup-participant.entity.ts:6-19](backend/src/cleanup/cleanup-participant.entity.ts#L6-L19)) keyed by `(cleanup_id, user_id)` with `role: 'member' | 'organizer'`. `CleanupDate` has `start_at`/`end_at`/`cleanup_id`/lat-lng ([cleanup-date.entity.ts](backend/src/cleanup/cleanup-date.entity.ts)). No `team_id` on either entity, so `teamId`/`pickedUp` filters don't apply to the cleanup-date layer.

`CleanupList.tsx` already implements the "My = participant" semantic ([CleanupList.tsx:111](frontend/src/components/CleanupList.tsx#L111)) — both `member` and `organizer` count.

## Implementation Plan

Backend-only. All edits in [backend/src/insights/insights.service.ts](backend/src/insights/insights.service.ts). No frontend, schema, or API-shape changes.

### Step 1 — `getMapData()` cleanup-rows query (lines 101-111)

Constrain the cleanup-date rows themselves by every filter that maps to that table:

| Filter | Cleanup-side predicate |
|--------|------------------------|
| `filter.userId` | `JOIN cleanup_participants cp ON cp.cleanup_id = c.id AND cp.user_id = $N` (inner join — both roles count) |
| `filter.since` | `cd.start_at >= $N` |
| `filter.cleanupDateId` | `cd.id = $N` |
| `filter.teamId` | not applied — no team relation |
| `filter.pickedUp` | not applied — boolean lives on spots only |

Introduce a `buildCleanupWhere(filter)` helper next to `buildSpotWhere`/`buildSpotAnd` returning `{ joins, where, params }`. Cleanup query becomes:

```sql
SELECT cd.id, cd.longitude, cd.latitude, cd.location_name,
       c.name AS cleanup_name, cd.start_at,
       COUNT(s.id)::int AS spot_count
FROM cleanup_dates cd
JOIN cleanups c ON c.id = cd.cleanup_id AND c.archived_at IS NULL
${cleanupJoins}            -- e.g. JOIN cleanup_participants cp ...
LEFT JOIN spots s ON s.cleanup_date_id = cd.id ${spotAnd}
${cleanupWhere}            -- merges existing "cd.latitude IS NOT NULL" with new conditions
GROUP BY cd.id, cd.longitude, cd.latitude, cd.location_name, c.name, cd.start_at
```

Notes:
- Two parameter sources (spot-side `andParams` + cleanup-side params). Either renumber `$N` at template time, or have `buildCleanupWhere` accept a starting index. Pick whichever reads cleaner.
- `cd.latitude IS NOT NULL` stays as baseline; new conditions are AND-appended.
- `spot_count` aggregate continues to use spot-side `${and}` so under "My" it shows the user's pick count per their cleanup — consistent with the spot layer.
- Cache key already includes `u:`, `t:`, `cd:`, `s:`, `pu:` ([line 172-180](backend/src/insights/insights.service.ts#L172-L180)) — no cache-key change.
- `buildSpotWhere`/`buildSpotAnd` untouched.

### Step 2 — `computeFilteredStats()` cleanup queries (lines 352-356, 390-396)

When `filter.userId` is set, replace the spot-derived cleanup queries with participant-derived ones (reusing `buildCleanupWhere`):

- **`totalCleanups`**:
  ```sql
  SELECT COUNT(DISTINCT c.id)::int AS count
  FROM cleanups c
  JOIN cleanup_participants cp ON cp.cleanup_id = c.id AND cp.user_id = $N
  WHERE c.archived_at IS NULL
    ${optional EXISTS clause for cleanupDateId / since via cleanup_dates}
  ```
  When date filters are present, gate via `EXISTS (SELECT 1 FROM cleanup_dates cd WHERE cd.cleanup_id = c.id AND cd.start_at >= $N [AND cd.id = $N])`.

- **`cleanupsTimeSeries`**:
  ```sql
  SELECT TO_CHAR(date_trunc('week', cd.start_at), 'YYYY-MM-DD') AS week,
         COUNT(DISTINCT c.id)::int AS count
  FROM cleanup_participants cp
  JOIN cleanups c ON c.id = cp.cleanup_id AND c.archived_at IS NULL
  JOIN cleanup_dates cd ON cd.cleanup_id = c.id
  WHERE cp.user_id = $N
    [AND cd.start_at >= $N]
    [AND cd.id = $N]
  GROUP BY date_trunc('week', cd.start_at)
  ORDER BY date_trunc('week', cd.start_at)
  ```
  Buckets weeks by `cd.start_at` ("weeks I had cleanup activity scheduled") instead of by spot capture week.

When `filter.userId` is **not** set, leave existing spot-derived queries untouched — for `teamId`/`cleanupDateId`-only filters, "cleanups with picks" remains the right semantic.

`getPublicStats`'s cache key already keys off `u:`, `t:`, `cd:`, etc. ([line 162-170](backend/src/insights/insights.service.ts#L162-L170)) — no cache-key change.

### Out of scope

- `/cleanups` list-page "My" filter (already correct).
- `computeGlobalStats` (no `userId` path).
- Non-cleanup `/insights` stats — correctly spot-derived under "My".
- Organizer-only sub-filter; both roles count as "mine".
- Frontend changes — none required.

## Verification

1. **SQL sanity:** with no filters, generated SQL is byte-identical to today. Each filter (`userId`, `since`, `cleanupDateId`) added in isolation adds exactly one predicate/join + one parameter.
2. **`/map` end-to-end** (user participates in cleanup A but not cleanup B, both with lat/lng):
   - "My" off → both A and B visible. "My" on → only A.
   - A.start_at < 7d ago, B.start_at > 30d ago. Preset "7d" → only A. "all" → both.
   - Active cleanup-date context = A → only A's marker.
   - "My" + "7d" intersects correctly.
3. **`/insights` end-to-end** (same user, plus cleanup C the user joined but logged no spots in):
   - "My" off → "Cleanups" headline shows global count.
   - "My" on → headline equals participant cleanup count (must include C).
   - "My" on + "7d" → drops to participant cleanups whose dates fall in last 7 days.
   - Cleanups bar chart under "My" buckets weeks by `cd.start_at`, not spot capture week.
4. **Cache:** flush Redis (or wait `cacheTtlSeconds`) between toggles when verifying.
5. **Regression:** `/cleanups` list and unfiltered (no-`userId`) `/insights` and `/map` render unchanged. Spot markers and all non-cleanup `/insights` stats behave identically under every filter combination.
