## Analysis
The "my picks" list is a hard-capped 50 items.

- Frontend [frontend/src/components/HistoryPanel.tsx:190-201](../../../frontend/src/components/HistoryPanel.tsx) hardcodes `limit: '50'` with no offset, cursor, or load-more UI.
- Backend [backend/src/spot/spot.controller.ts:272-303](../../../backend/src/spot/spot.controller.ts) `GET /spots` accepts `limit` (default 20, max 100) but **no offset/cursor**. The service [backend/src/spot/spot.service.ts:295-320](../../../backend/src/spot/spot.service.ts) calls `.take(limit)` ordered by `captured_at DESC` with no `.skip()`.

Result: anything older than the 50th most recent pick is unreachable. Once cleanups grow, this gets worse.

## Implementation Plan
1. **Backend — add cursor pagination** (preferred over offset for stable order under inserts).
   - In [backend/src/spot/spot.service.ts](../../../backend/src/spot/spot.service.ts) `listSpotsWithFilters`: accept an optional `before: string` (a `captured_at` ISO timestamp + id tiebreaker). Apply `WHERE (captured_at, id) < (:beforeAt, :beforeId)` to the existing query.
   - In [backend/src/spot/spot.controller.ts](../../../backend/src/spot/spot.controller.ts): add `before` query param, pass through.
   - Response shape: `{ items, nextCursor }` where `nextCursor` is the last item's `captured_at|id` joined, or `null` when fewer than `limit` returned.
2. **Frontend — load-more button.**
   - Track `cursor` in [HistoryPanel.tsx](../../../frontend/src/components/HistoryPanel.tsx) state. On first load, fetch with no cursor. Render a "Load more" button when `nextCursor !== null`. Append, don't replace.
   - Keep the visible page size around 50 to avoid huge DOM.
3. **Tests** — add a backend test covering cursor consistency across two pages with an insert in between.
