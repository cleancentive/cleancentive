## Analysis
The "asleep" state is the existing `failed` state in [frontend/src/lib/pendingPicks.ts](../../../frontend/src/lib/pendingPicks.ts) — picks transition `pending → uploading → failed`, after which they sit waiting for `nextRetryAt` (exponential backoff up to 15 min for 5xx; 24 h for 4xx except 429).

A 502 specifically indicates the upstream isn't responding to the proxy. Likely culprits:

1. **Caddy → backend timeout/size limits.** [infrastructure/caddy/Caddyfile:4-6](../../../infrastructure/caddy/Caddyfile) has no explicit `request_body` size or `transport http { response_header_timeout … }` configured. With ~7 images at up to 15 MB each (`UPLOAD_MAX_SIZE_BYTES`), defaults can drop the upstream connection.
2. **Backend single-request multipart processing blocks the event loop.** Look at [backend/src/spot/spot.controller.ts:164-176](../../../backend/src/spot/spot.controller.ts) — `FileFieldsInterceptor` handles uploads in-process; large or concurrent batches can starve and time out.
3. **Frontend treats 502 as transient and applies long backoff.** Picks "fall asleep" because the retry timer pushes them out 1–15 min, with no visible "stuck" state.

The diagnostics added in commit 27e9ae2 already report ≥3 failed attempts to `/api/v1/client-events`. We should now have prod telemetry — check it before code changes.

## Implementation Plan
1. **Pull telemetry first.** Query [backend/src/client-events](../../../backend/src/client-events) data for `pick_upload_failed` in the last 14 days. Look for status-code distribution, sizes, user agents. This tells us whether the 502 is rare (proxy hiccup) or systemic (limit).
2. **Caddy config — raise/explicit limits.**
   - Add `request_body { max_size 200MB }` (or whatever budget covers the worst-case batch).
   - Add an explicit `reverse_proxy` timeout block: `transport http { read_timeout 120s write_timeout 120s }`.
3. **Backend — verify multer/Nest limits match Caddy.** Confirm `UPLOAD_MAX_SIZE_BYTES` × per-request file count isn't capped lower than Caddy. Add a `payload too large` 413 with a clear message instead of letting the request stall.
4. **Frontend — better visibility for stuck picks.**
   - In [frontend/src/lib/pendingPicks.ts](../../../frontend/src/lib/pendingPicks.ts) and the picks list UI, show a clear "Retrying in N min — tap to retry now" affordance when `failed` with a future `nextRetryAt`. Today they appear "asleep" because they're silent.
   - On 502 specifically, retry sooner (e.g., 30 s × 2 attempts before falling into the long backoff).
5. **Reach out to reporter** once a fix is shipped, asking them to retry and confirm.
6. **Cross-link** with [docs/TODO_file-outline-bug.md](../../../docs/TODO_file-outline-bug.md) — that document tracks the broader upload-stuck investigation; this report is one symptom.
